# Design Doc: Math Genie 比较练习重构（信息架构 + 应用题补全）

## 概述

本 doc 基于实际代码现状（`SpecialPracticeType.COMPARISON`、
`ComparisonSubType`、`generators/comparison.ts` 等，详见下方引用）
重新设计比较练习功能，分两部分：

1. **信息架构重构**：将"比较练习"从"加减法练习的专项练习选项之一"
   提升为与"加减法练习"平级的题型大类，采用两步式选择（先选大类，
   再展开对应详细配置），解决当前"专项练习"五个选项语义不对等、
   选中比较练习后大量加减法专属字段隐藏/失效导致困惑的问题。

2. **应用题生成逻辑补全**：当前比较练习的"应用题"呈现方式是空壳
   （选择后实际渲染与文本模式完全一致，无独立文字生成），本次
   需要补齐真实的应用题文字生成逻辑。

**本 doc 取代此前基于截图猜测的版本**，所有结构判断已对照实际代码
确认，过时或被推翻的判断不再保留。

## 1. 现状基线（已确认的真实代码行为）

### 1.1 类型定义

```typescript
// src/types.ts
export enum SpecialPracticeType {
  NONE = 'none',
  ZERO_DRILL = 'zero_drill',
  FACT_FAMILY = 'fact_family',
  NUMBER_BOND = 'number_bond',
  COMPARISON = 'comparison',
}

export enum ComparisonSubType {
  MAGNITUDE = 'magnitude',   // 比大小
  DIFFERENCE = 'difference', // 比多少
}

// Config 字段
comparisonConfig?: { subType: ComparisonSubType };
```

> 注：`WORD_PROBLEM_COMPARISON` 已在此前的"比较练习 v3 重构"中被
> 移除，原应用题比多少逻辑理论上应合并到 `COMPARISON +
> DisplayMode.WORD_PROBLEM` 路径下，但实际只完成了路由层面的合并，
> 未完成真正的文字生成逻辑（见 §1.3）。

### 1.2 两个子题型的真实区别

两者共享同一个数据结构 `ComparisonData`，唯一区别是 `allowEqual`：

| | MAGNITUDE（比大小） | DIFFERENCE（比多少） |
|---|---|---|
| 允许相等 | ✅ | ❌（生成时循环重试直到不等） |
| 渲染 | 圆圈填 `>`/`<`/`=` | 文字"X比Y多/少___个"+填空数字 |

### 1.3 呈现方式覆盖矩阵（含已确认的空壳问题）

| | 纯数字 | 图案 | 应用题 |
|---|---|---|---|
| 比大小 | ✅ 已实现 | ✅ 已实现 | 🚫 互斥禁用（产品逻辑：比大小做应用题无意义） |
| 比多少 | ✅ 已实现 | ✅ 已实现 | ⚠️ **空壳**：选择后强制 subType→DIFFERENCE，但渲染退化为纯文本，无应用题文字生成 |

**比大小×应用题互斥是既有产品逻辑**（代码中 `disabled={displayMode
=== DisplayMode.WORD_PROBLEM}`，理由：比大小的应用题形式在产品
逻辑上不成立，只有比多少才适合应用题场景）。本次重构**保留这条
互斥规则**，不改变。

### 1.4 难度配置复用情况

- `difficulty` / `customDifficulty`：比较练习完全复用加减法的字段，
  不新建独立的数值范围配置
- `difficultyRatios`（混合难度）：**当前对比较练习不生效**——混合
  模式的比率逻辑只在 `FILL_BLANK` 和标准题路径中处理，比较练习
  选了"混合"tab 不会有任何效果
- EMOJI 模式特殊限制：图案呈现 + 比较练习时，若数值范围 max>10，
  会弹出确认提示（"数字超过10在图案模式下会显示不全"），确认后
  自动 cap 到 `{min:1, max:10}`

### 1.5 切换状态保留规则

| 切换方向 | `difficulty` | `comparisonConfig` |
|---|---|---|
| 加减法 → 比较练习 | 保留 | 新建为 `{subType: MAGNITUDE}`（不恢复用户上次在比较模式下的选择） |
| 比较练习 → 加减法 | 保留 | 清空（`undefined`） |

### 1.6 控件可见性规则

```typescript
const hideArithmeticControls = isComparisonSelected || isNumberBond;
```

为 `true` 时：运算类型、题目类型选择器**隐藏**（不是禁用变灰，是
直接不渲染）。显示模式、专项练习、难度/范围控件**始终可见**。

## 2. 信息架构重构方案

### 2.1 两步式结构

```
Step 1：题型大类选择器（新增组件，置于设置面板最顶部）
  ├─ 加减法练习
  └─ 比较练习

Step 2：根据 Step 1 选择，展开对应的独立配置卡片组
  ├─ 加减法练习 → 现有的"页面与输出""题目与难度""过滤规则"卡片
  │   （从专项练习选项中移除 COMPARISON，只保留
  │    无/零/数组家族/数的分合）
  └─ 比较练习 → 新的卡片组（见 2.2）
```

**关键修正**（不同于此前原型版本）：

- 此前原型把"应用题"作为比较练习呈现方式的平级第三选项展示，
  但当前它是空壳。本次方案是**先补完应用题生成逻辑（见 §3），
  UI 上才正式展示三个平级选项**，不是先做 UI 再留一个不工作的
  按钮在那里——这一点已与产品侧确认（选择是"补完整"，不是"暂时
  隐藏"或"加说明文字搪塞"）
- 呈现方式选择器需要体现"比大小×应用题互斥"：当用户已选择"比大小"
  子题型时，"应用题"呈现选项应禁用并附 tooltip 说明（"比大小暂不
  支持应用题形式，请切换到比多少"）；反之，当用户已选择"应用题"
  呈现方式时，"比大小"子题型选项应禁用

### 2.2 比较练习的独立卡片组

```
卡片「比较题型」
  - 子题型选择：比大小 / 比多少（ToggleButtonGroup，身份色=math-genie蓝）
  - 呈现方式：纯数字 / 图案 / 应用题（三选一，含上述互斥逻辑）

卡片「数值与难度」
  - 数值范围：1-5 / 1-10 / 1-20 / 自定义
    （复用 difficulty/customDifficulty，与加减法共享同一份状态，
    这是已确认的现状，本次不改变这个复用关系）
  - 不展示"单一/混合"tab——因为 difficultyRatios 对比较练习不生效，
    展示一个不工作的控件比不展示更糟（参照此前 UX audit 原则：
    不展示用户会误以为有效的失效控件）

卡片「页面与输出」
  - 页数等现有字段保留
```

**关于"单一/混合"tab 的处理**：现状是混合难度对比较练习完全不
生效，但 UI 上没有任何提示，这是一个新发现的隐藏陷阱（不属于此前
任何一轮 audit 发现的问题，但同属于"用户操作了某个控件，却没有
对应效果"这一类问题）。本次方案选择**直接不展示**这个 tab，而非
展示后禁用+解释——因为"混合难度"这个概念对比较练习在产品逻辑上
本身就不成立（比较练习不是按难度级别出题，是按数值范围出题），
不需要保留一个本来就不该存在的选项再去解释为什么它不工作。

### 2.3 EMOJI 模式数值范围确认弹窗

沿用现状已有的确认弹窗逻辑（"数字超过10在图案模式下会显示不全"），
本次不修改这个交互，仅确保两步式重构后这个弹窗仍在正确的时机触发
（即用户在比较练习卡片组内，将呈现方式切到"图案"且数值范围>10时）。

## 3. 应用题生成逻辑补全（新增开发任务）

### 3.1 任务范围

为"比多少"子题型实现真实的应用题文字生成，参照加减法练习现有的
`generators/word-problem.ts` 的实现模式（模板化生成，参见设计文档
中该文件的职责描述："Contextual word problems from templates"）。

**重要提醒**：本 doc 不直接照抄设计文档里对 `word-problem.ts` 的
概括性描述去设计新函数——那份描述本身可能和实际实现有细节差异
（参考本次信息收集过程中已经发生过的"设计文档描述与实际代码不
一致"情况）。实现前请先实际阅读 `generators/word-problem.ts` 的
源码，确认其模板结构、变量替换方式、emoji/主题注入方式，再设计
比较练习的应用题生成函数，保持两者风格一致。

### 3.2 预期生成效果（示例，非最终文案）

参照 `ComparisonData` 现有字段（`groupA`、`groupB`、`relation`、
`difference`），应用题文字生成大致应包含：

```
"小明有 5 个苹果🍎，小红有 3 个苹果🍎，小明比小红多几个？"
（对应 groupA.count=5, groupB.count=3, difference=2）
```

具体人物/场景模板措辞、是否需要多套模板轮换避免重复感，请参照
`word-problem.ts` 现有加减法应用题的模板设计风格决定，本 doc 不
预先规定具体文案。

### 3.3 与现有数据结构的衔接

新生成函数应直接消费现有 `ComparisonData` 结构（`groupA`/`groupB`/
`difference`），不需要重新设计数据结构，只需新增一个"将 `ComparisonData`
转换为应用题文字"的渲染/生成函数，建议命名和文件组织参照现有
`generators/comparison.ts` 的同级目录结构，例如新增
`generators/comparison-word-problem.ts` 或在 `comparison.ts` 内
新增一个导出函数，具体由实现时根据现有文件大小和组织习惯决定。

### 3.4 完成后需要解除的限制

完成应用题生成逻辑后：

1. 移除"应用题"渲染退化为文本模式的临时逻辑
2. 确认"比大小×应用题"互斥逻辑保留（这不是因为应用题没实现，
   是因为产品逻辑上比大小本身不适合应用题形式，与本次补全任务
   无关，补全后这条互斥规则依然要存在）
3. 更新 §2.1 提到的 UI 三选一呈现方式选择器，正式上线

## 4. 实现阶段划分

| Phase | 内容 | 依赖 |
|---|---|---|
| 1 | 应用题生成逻辑补全（§3） | 需先读 `word-problem.ts` 源码 |
| 2 | 两步式信息架构重构：Step 1 大类选择器组件 | 无，可与 Phase 1 并行 |
| 3 | 比较练习独立卡片组（§2.2），含呈现方式三选一+互斥逻辑 | 依赖 Phase 1（否则三选一只能先做两个）和 Phase 2 |
| 4 | 加减法练习卡片组调整（移除专项练习中的比较练习选项） | 依赖 Phase 2 |

**建议顺序**：Phase 1 和 Phase 2 并行开始；Phase 1 完成后才能让
Phase 3 的呈现方式选择器三个选项都真正生效（否则会重新出现"选了
没反应"的问题，等于重构了 UI 但没解决根本问题）。

## 5. 验收标准

- [ ] 设置面板最顶部出现题型大类选择器，加减法练习/比较练习视觉
      权重相同、并列展示
- [ ] 加减法练习的专项练习选项中不再包含"比较练习"
- [ ] 比较练习有独立的卡片组，不与加减法练习的运算类型/题目类型
      等字段混在一起
- [ ] 比较练习卡片组不展示"单一/混合"难度 tab
- [ ] 应用题呈现方式选中"比多少"时，生成真实的应用题文字（非
      退化为纯文本格式），且文案风格与加减法应用题模板风格一致
- [ ] "比大小"子题型选中应用题呈现方式时，按钮禁用并有说明；
      反之"应用题"呈现方式选中比大小子题型时同理
- [ ] 切换题型大类（加减法↔比较）时，`difficulty` 保留，
      `comparisonConfig` 按现有规则处理（此次不改变这个规则，
      只是确认两步式 UI 下这个行为依然正确）
- [ ] EMOJI 模式 + 比较练习 + 数值范围>10 的确认弹窗逻辑在新 UI
      结构下依然正确触发

---

## 6. 本次明确不处理的范围

- 不重新设计 `difficultyRatios` 混合难度逻辑使其对比较练习生效
  ——本次选择是"不展示这个选项"，而非"让它工作"，如果后续产品
  决定比较练习也需要混合难度概念，需要单独立项设计（这涉及"按
  数值范围混合"这个概念在比较类题目里如何定义，不是简单复用加减法
  的混合逻辑）
- 不改变"比大小×应用题互斥"这条已有产品规则
- 不涉及加减法练习本身的任何功能改动，只移除其专项练习选项中的
  "比较练习"这一项
